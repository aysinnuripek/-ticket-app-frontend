1. Project Overview
Product. A web application where end users browse upcoming events (concerts, sports, theatre, festivals), view event details, select tickets, pay online with a credit card, and receive a digital ticket containing a QR code via email. Event organizers can create events and view their sales through an admin panel.
Why this is a strong cloud project. Ticket sales have very spiky traffic patterns: when a popular event goes on sale, hundreds of users hit the system in a few minutes, then the load drops off again. This kind of bursty workload is a perfect excuse to use auto-scaling containers, load balancers, asynchronous queues, in-memory caches, and managed databases — all of which give you concrete, justifiable design decisions for the project report.
Core user flows.
	•	Browse events: open the homepage, see a list of upcoming events filtered by city and category.
	•	View event details: see seat availability, prices, venue info, and event description.
	•	Purchase: sign in, select seats/tickets, pay via Stripe (test mode), receive an email with a QR code ticket.
	•	My tickets: view past purchases and re-download the QR code.
	•	Admin: organizers can create events, set prices, and view sales reports.

2. Proposed Architecture (High-Level Design)
The architecture follows a standard three-tier pattern (client, application, data) extended with asynchronous workers for slow tasks. Everything runs inside a single AWS Virtual Private Cloud (VPC) spanning two Availability Zones for high availability.
Layer-by-Layer Breakdown
Frontend (Client)
	•	React single-page application built with Vite.
	•	Hosted on Amazon S3 (static files) and served globally through Amazon CloudFront (CDN) for low latency anywhere in the world.
Backend (API)
	•	An Application Load Balancer (ALB) receives HTTPS traffic and distributes it across multiple containers.
	•	The API runs on Amazon ECS Fargate — serverless containers, so we do not manage EC2 servers.
	•	Auto-scaling: the number of running containers grows when CPU or request count rises, and shrinks again when traffic drops.
	•	Built with Python and FastAPI, packaged as a Docker image.
Data Layer
	•	Amazon RDS (PostgreSQL) — relational database for users, events, orders, and tickets. Deployed Multi-AZ so a single AZ failure does not bring the system down.
	•	Amazon ElastiCache (Redis) — used for two things: temporarily locking seats during checkout (10-minute TTL) so two users cannot buy the same seat, and caching the popular event list to reduce database load.
Async / Spike Handling
	•	Amazon SQS — when a user successfully pays, the API drops a message on a queue instead of doing the slow work inline. The user gets an instant response.
	•	AWS Lambda — picks up the queue message, generates the QR code, builds a PDF ticket, uploads it to S3, and sends the confirmation email.
	•	Amazon SES — sends transactional emails (ticket delivery, order confirmation).
Authentication
	•	Amazon Cognito — handles signup, login, password reset, and issues JWT tokens. Saves us from building (and securing) auth from scratch.
Storage
	•	Amazon S3 — event banner images and generated QR-code PDF tickets.
Networking
	•	A VPC with public subnets (ALB, NAT Gateway) and private subnets (ECS tasks, RDS, ElastiCache).
	•	Amazon Route 53 for DNS, AWS Certificate Manager (ACM) for free HTTPS certificates.
Observability & IaC
	•	Amazon CloudWatch — centralized logs, metrics, and alarms.
	•	Terraform — everything above is defined as code so it is reproducible and easy to document in the report.
System-Wide Availability Decisions
	•	RDS is deployed Multi-AZ: AWS keeps a hot standby in a second Availability Zone.
	•	ECS tasks are spread across two AZs by the ALB, with a minimum of 2 running tasks at all times.
	•	CloudFront caches static assets at edge locations, reducing origin load.
	•	Slow work (emails, PDF generation) is decoupled through SQS so a backed-up worker never blocks user requests.
	•	Health checks on the ALB automatically remove unhealthy containers and replace them.

Note for the report: in the report, this section should be accompanied by a clear architecture diagram. Use a free tool like draw.io, Lucidchart, or AWS’s own architecture icons to draw one. Show users → CloudFront → ALB → ECS → RDS/Redis, plus the SQS → Lambda → SES side flow.

4. Database Schema (ERD)
Below is the minimum viable schema. Member B is responsible for refining and migrating it. Primary keys are UUIDs.

Tables
users
	•	id (UUID, PK)
	•	cognito_sub (string, unique) — the Cognito user ID.
	•	email, full_name, role (enum: customer | organizer | admin)
	•	created_at, updated_at
venues
	•	id (UUID, PK)
	•	name, city, address, capacity
events
	•	id (UUID, PK)
	•	title, description, category, image_url
	•	venue_id (FK → venues.id)
	•	organizer_id (FK → users.id)
	•	starts_at, ends_at, status (draft | published | cancelled)
	•	created_at, updated_at
ticket_types
	•	id (UUID, PK)
	•	event_id (FK → events.id)
	•	name (e.g. “General Admission”, “VIP”), price_cents, currency, total_quantity, sold_quantity
orders
	•	id (UUID, PK)
	•	user_id (FK → users.id)
	•	status (pending | paid | failed | cancelled)
	•	total_cents, currency, stripe_session_id, created_at
order_items
	•	id (UUID, PK)
	•	order_id (FK → orders.id)
	•	ticket_type_id (FK → ticket_types.id)
	•	quantity, unit_price_cents
tickets
	•	id (UUID, PK) — used inside the QR code.
	•	order_id (FK → orders.id)
	•	ticket_type_id (FK → ticket_types.id)
	•	qr_code_url (S3 link), used (boolean)

Members are referred to as A, B, C, D — the team can map these to real names in the first meeting.

6. Detailed Implementation Guidelines
This section gives each member a step-by-step “do this, then that” guide. The goal is to remove ambiguity so each person can pick up their part and start coding immediately. Steps refer to AWS services using their official names so they are easy to look up in the AWS console or in Terraform documentation.
6.1 Member A — Step-by-Step Guide
Task A1: VPC & Networking Setup
Goal: create a VPC where private resources (database, containers) are unreachable from the internet directly.
	•	Create a VPC with CIDR block 10.0.0.0/16.
	•	Create 4 subnets across two AZs (e.g. eu-central-1a and eu-central-1b): 2 public (10.0.1.0/24, 10.0.2.0/24) and 2 private (10.0.11.0/24, 10.0.12.0/24).
	•	Create an Internet Gateway and attach it to the VPC.
	•	Create a NAT Gateway in one of the public subnets and allocate an Elastic IP for it. This lets private resources reach the internet (for OS updates, calling Stripe, etc.) without being reachable themselves.
	•	Create route tables: one for the public subnets (default route to the Internet Gateway) and one for the private subnets (default route to the NAT Gateway).
	•	Create security groups:
	•	ALB SG — allow inbound 80/443 from the internet.
	•	ECS SG — allow inbound traffic only from the ALB SG on the app port (e.g. 8000).
	•	RDS SG — allow inbound 5432 only from the ECS SG.
	•	Redis SG — allow inbound 6379 only from the ECS SG.
Deliverable: a Terraform module `infra/network/` that recreates the entire VPC with a single `terraform apply`.
Task A2: ECS Fargate Cluster
Goal: have a serverless container platform that runs the FastAPI backend behind a load balancer.
	•	Create an ECR repository named `ticket-app-backend`. This is where Docker images go.
	•	Create an ECS cluster of type Fargate. No EC2 instances to manage.
	•	Define a task definition with: 0.5 vCPU, 1 GB memory, container port 8000, image URL from ECR, environment variables (DB host, Redis host, SQS queue URL, Cognito user pool ID), and an execution role that allows pulling from ECR and writing to CloudWatch Logs.
	•	Create an Application Load Balancer in the public subnets with an HTTPS listener using the ACM certificate (Task A6).
	•	Create a target group of type `ip` with health-check path `/health` (Member B will implement this endpoint to return 200 OK).
	•	Create an ECS service that links the task definition to the target group, running in the private subnets, with desired count = 2.
	•	Configure auto-scaling on the service: min 2, max 6, target tracking on average CPU > 60%.
Deliverable: Terraform module `infra/ecs/` and a working URL (the ALB DNS name) that returns 200 on `/health`.
Task A3: CI/CD Pipeline
Goal: every push to `main` automatically builds the backend, pushes it to ECR, and rolls out a new ECS deployment.
	•	Create an IAM user for GitHub Actions with permissions to push to ECR and update the ECS service (or use OIDC for keyless auth, which is the modern approach).
	•	Add the GitHub Actions secrets: AWS_ROLE_ARN, AWS_REGION, ECR_REPOSITORY, ECS_CLUSTER, ECS_SERVICE.
	•	Create `.github/workflows/deploy.yml` with steps: checkout → configure AWS credentials → login to ECR → docker build → docker push → `aws ecs update-service --force-new-deployment`.
	•	Add a pull-request workflow that runs the test suite but does not deploy. Member D will plug E2E tests into this.
Deliverable: a green pipeline run that visibly deploys a code change to production within ~5 minutes.
Task A4: Terraform Codebase
Goal: every AWS resource the project uses is described as code, not clicked together in the console.
	•	Set up a Terraform project structure: `infra/` with modules `network/`, `ecs/`, `rds/`, `redis/`, `s3/`, `cognito/`, `sqs/`, `lambda/`, `cloudfront/`.
	•	Use a remote backend (S3 + DynamoDB lock table) so the team shares state safely.
	•	Tag every resource with `Project=TicketApp` and `Owner=<member-name>` for cost tracking.
	•	Document each module with a short README explaining inputs, outputs, and intent. These READMEs feed directly into the report.
Deliverable: running `terraform apply` from a clean state recreates the entire infrastructure.
Task A5: CloudWatch Monitoring
Goal: see the system's behavior at a glance and get alerted when something breaks.
	•	Enable CloudWatch Container Insights on the ECS cluster (one-click setting).
	•	Create a CloudWatch Dashboard with widgets for: ALB request count, ALB 5xx error rate, ECS CPU utilization, RDS CPU, RDS connection count, SQS queue depth, Lambda invocation count and errors.
	•	Create alarms: 5xx errors > 1% for 5 minutes, RDS CPU > 80% for 10 minutes, SQS queue depth > 100 for 5 minutes, Lambda errors > 0.
	•	Send all alarms to an SNS topic and subscribe the team's email addresses.
Deliverable: screenshots of the dashboard under load (Member D's load test) for the report.
Task A6: Route 53 + ACM
	•	Register a cheap domain (~$10/year) or use a subdomain of one a team member already owns.
	•	Create a hosted zone in Route 53.
	•	Request a wildcard certificate `*.example.com` via ACM (free). Use DNS validation.
	•	Create A/AAAA alias records: `api.example.com` → ALB, `www.example.com` → CloudFront.
6.2 Member B — Step-by-Step Guide
Task B1: Database Schema
	•	Install dependencies: `pip install fastapi uvicorn sqlalchemy alembic psycopg2-binary python-dotenv`.
	•	Create `app/models/` with one file per table (`user.py`, `event.py`, `order.py`, etc.) based on the schema in Section 4.

	•	Seed the DB with 10 sample events for the frontend team to develop against.
Deliverable: a runnable local environment + a seed script. Hand the seed script to Member C so they have data to display.
Task B2: REST API Endpoints
Build these endpoints in FastAPI. Define Pydantic models for every request/response so the OpenAPI docs at `/docs` are auto-generated and complete.

Public endpoints:
	•	GET /health — used by the ALB; returns 200 OK.
	•	GET /events?city=...&category=... — list published events.
	•	GET /events/{id} — single event with ticket types.
Authenticated user endpoints:
	•	POST /orders — body: list of {ticket_type_id, quantity}. Reserves seats in Redis (Task B3), creates a pending Order, returns a Stripe Checkout session URL.
	•	GET /orders/me — list the current user's orders.
	•	GET /tickets/{id} — returns ticket detail and the S3 link to the QR PDF (if generated).
Organizer / admin endpoints:
	•	POST /admin/events — create an event.
	•	PUT /admin/events/{id} — update an event.
	•	GET /admin/events/{id}/sales — sales summary.
Webhook (called by Stripe, not the user):
	•	POST /webhooks/stripe — receives Stripe events. Member D will help wire this up.
Deliverable: OpenAPI spec at /docs. Lock the spec in Week 2 so Member C can build against it with confidence.
Task B3: Seat / Ticket Locking with Redis
Problem: two users click “buy” on the last ticket at the same time. Without a lock, both purchases would succeed and one customer gets a refund and a bad experience.
	•	Install redis-py: `pip install redis`.
	•	On `POST /orders`, before creating the order, try to acquire a Redis key per ticket_type: `SET lock:ticket_type:{id}:{order_id} 1 NX EX 600` for each unit being reserved. If any SET fails (because someone else already holds the lock), reject the order with HTTP 409.
	•	Decrement an `available:{ticket_type_id}` counter atomically using `DECRBY`. Reject if it would go below zero (reincrement and return 409).
	•	If the user pays in time, the webhook handler commits the order to the DB and the locks expire naturally.
	•	If the user does not pay in time, the locks expire after 10 minutes and the tickets become buyable again.
Deliverable: a test (with pytest + fakeredis) showing that two concurrent orders for the same last ticket only succeed once.
Task B4: SQS Producer Logic
	•	Member A creates an SQS queue called `payment-success`. Get its URL into your environment variables as SQS_QUEUE_URL.
	•	Install boto3: `pip install boto3`.
	•	In the Stripe webhook handler, on a `checkout.session.completed` event: update the order status to `paid` in the DB, then call `sqs_client.send_message(QueueUrl=..., MessageBody=json.dumps({'order_id': str(order.id)}))`.
	•	Use IAM Task Role permissions — do not put AWS keys in the code.
Deliverable: a test order whose successful payment triggers a visible SQS message.
Task B5: Cognito Integration
	•	Member A provisions a Cognito User Pool and App Client; get the User Pool ID and the JWKS URL.
	•	Install python-jose: `pip install "python-jose[cryptography]"`.
	•	Create a FastAPI dependency `get_current_user` that: reads the Bearer token from the Authorization header, fetches the JWKS (cache it), verifies the signature and `aud`/`iss` claims, and returns the user from the DB (creating one on first sign-in based on the `sub` claim).
	•	Apply the dependency to every authenticated route. Organizer routes also check `role == 'organizer'` or `'admin'`.
Deliverable: an authenticated `GET /me` endpoint that returns the user's profile when given a Cognito JWT.
Task B6: Dockerize the Backend
Sample multi-stage Dockerfile to copy:
FROM python:3.11-slim AS base
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
	•	Add a `.dockerignore` (exclude .venv, __pycache__, .git, tests).
	•	Test locally: `docker build -t ticket-app . && docker run -p 8000:8000 ticket-app`. Verify `/health` returns 200.
	•	Hand off to Member A: the image name and the port (8000).

6.4 Member D — Step-by-Step Guide
Task D1: Lambda Consumer for SQS
	•	Create a Python 3.11 Lambda function called `process-payment`.
	•	Configure SQS as the event source (Member A wires this in Terraform).
	•	Lambda environment variables: DATABASE_URL (read-only is fine), TICKETS_BUCKET, SES_FROM_ADDRESS.
	•	Function logic in pseudocode:
for record in event['Records']:
    order_id = json.loads(record['body'])['order_id']
    order = fetch_order(order_id)
    for ticket in order.tickets:
        qr_png = generate_qr(ticket.id)        # using 'qrcode' library
        pdf = build_pdf(ticket, qr_png)          # using 'reportlab'
        s3.put_object(Bucket=..., Key=f'tickets/{ticket.id}.pdf', Body=pdf)
    send_ses_email(order.user.email, tickets=order.tickets)
	•	Build dependencies as a Lambda Layer (zip + upload) since `reportlab` is too large to bundle inline.
	•	Set the Lambda role to allow: read SQS, write S3 (only the tickets bucket), send SES, read RDS (via VPC config).
	•	Test by sending a fake message to the queue.
Deliverable: a PDF ticket with QR code arriving in the test inbox within 30 seconds of payment.

	•	Test rendering in Gmail, Outlook web, and Apple Mail (they each render HTML differently).
Task D3: Stripe Backend Integration
	•	Create a Stripe account (free) and grab the test mode publishable key + secret key. Put the secret in AWS Secrets Manager; share only the publishable key with Member C.
	•	On `POST /orders` (this code lives in Member B's backend, but you'll write the Stripe portion), call `stripe.checkout.Session.create(...)` with the line items and your success/cancel URLs.
	•	Save the `session.id` on the order row so the webhook can look it up later.
	•	Implement `POST /webhooks/stripe`: read the raw body, call `stripe.Webhook.construct_event(payload, sig_header, webhook_secret)`. On `checkout.session.completed`, mark the order paid and trigger the SQS message (Task B4).
	•	Use Stripe CLI in dev to forward webhooks to localhost: `stripe listen --forward-to localhost:8000/webhooks/stripe`.
Task D4: End-to-End Testing
	•	Install Playwright: `npm init playwright@latest` inside `frontend/`.
	•	Write one comprehensive happy-path test: visit homepage → click an event → select 2 GA tickets → sign in (or sign up with a fresh email) → pay with Stripe test card → land on success page → see ticket in My Tickets.
	•	Add 3–5 smaller tests: invalid login, viewing My Tickets while signed out (should redirect), trying to buy more tickets than available.
	•	Run tests in GitHub Actions (Member A's PR workflow) against a deployed staging URL, not localhost.
Task D5: Load Testing
	•	Install k6 locally (`brew install k6` or download from k6.io).
	•	Write `loadtest/browse.js`: ramp up to 500 virtual users over 2 minutes, each repeatedly hitting `GET /events` and `GET /events/{id}`.
	•	Write `loadtest/checkout.js`: 50 virtual users running the full purchase flow. (Use Stripe test mode — no real charges.)
	•	Run both tests against the deployed staging environment.
	•	Capture: p95 response time, error rate, ECS task count over time (from CloudWatch), RDS CPU. These numbers and graphs are gold for the High Level Design / availability section of the report.
Task D6: Security Pass
	•	Audit IAM roles: every ECS task role and Lambda role should have the minimum permissions to do its job. No `*:*`.
	•	Confirm all secrets (Stripe keys, DB password) are in AWS Secrets Manager or SSM Parameter Store, never committed.
	•	Confirm the RDS security group only allows traffic from the ECS security group.
	•	Confirm only the frontend S3 bucket is publicly readable; tickets bucket must be private with signed URLs.
	•	Run `npm audit` and `pip-audit`; fix any high or critical CVEs.
	•	Enable AWS GuardDuty (free trial) for 7 days during the demo period — cheap and impressive in the report.

8. Coordination & Best Practices
Communication
	•	Weekly 30-minute sync — same day every week. Each person reports: what I did, what I will do, what is blocking me.
	•	Shared Notion / Trello / GitHub Projects board with columns: Backlog, In Progress, Blocked, Review, Done.
	•	A `#project` channel on Slack or Discord for quick questions — do not wait until the weekly sync.
Git Workflow
	•	One repository, with two top-level folders: `backend/` (Member B + D) and `frontend/` (Member C). Member A's `infra/` folder lives at the root.
	•	Feature branches: `feat/event-list`, `fix/login-error`. Never push directly to `main`.
	•	Pull requests require at least one approving review. Member A's PRs get reviewed by Member B; Member C and D review each other.
	•	Squash-merge into main.
API Contract First
	•	Member B publishes the OpenAPI spec in Week 2 and pins it. Changes after Week 4 must be communicated explicitly in the team channel.
	•	Member C develops against mock JSON in Week 2–3 to avoid being blocked while the backend is built.
Documentation Habits
	•	Every member keeps a running personal log of “what I chose and why” in their own markdown file.
	•	Take screenshots of AWS console configurations as you go — they are gold for the configuration sections of the report.
	•	Commit messages should describe the WHY, not just the WHAT.
Cost Hygiene
	•	Tag every AWS resource with `Project=TicketApp` and view the AWS Cost Explorer filtered on that tag weekly.
	•	Stop ECS services and the NAT Gateway overnight (they are the biggest cost) using a scheduled Lambda or just manually.
	•	Delete the entire stack via `terraform destroy` after the project is graded — do not leave it running.

10. Glossary
Quick definitions for the AWS services and other tools mentioned in this plan, in case anyone in the team is new to them.

Term
Meaning
ACM
AWS Certificate Manager. Issues free TLS/SSL certificates for HTTPS.
ALB
Application Load Balancer. Distributes HTTP/HTTPS traffic across multiple containers.
Alembic
Database migrations tool for SQLAlchemy.
Amplify
AWS frontend library; commonly used here only for its Cognito auth helpers.
AZ
Availability Zone. A physically isolated datacenter within an AWS Region.
CDN
Content Delivery Network. Caches static assets close to the user.
CloudFront
AWS’s CDN.
CloudWatch
AWS’s observability service: logs, metrics, alarms, dashboards.
Cognito
AWS’s managed user-identity service (signup, login, JWTs).
ECR
Elastic Container Registry. AWS’s Docker image registry.
ECS
Elastic Container Service. Runs Docker containers; Fargate is its serverless mode.
FastAPI
Modern, fast Python web framework with built-in OpenAPI docs.
IaC
Infrastructure as Code. Defining infra in version-controlled files (Terraform).
JWT
JSON Web Token. Signed user-identity token returned by Cognito.
Lambda
AWS’s serverless function service. Pay only for invocations.
NAT Gateway
Lets private-subnet resources reach the internet without being reachable.
RDS
Relational Database Service. AWS-managed PostgreSQL / MySQL / etc.
Redis
In-memory key/value store. Used here for seat locks and caching.
SES
Simple Email Service. AWS’s transactional email sender.
SQS
Simple Queue Service. AWS’s managed message queue.
TLS
Transport Layer Security. The protocol behind HTTPS.
VPC
Virtual Private Cloud. Your isolated network on AWS.
